      GERAM CORE OS · FIRST RUN
      
              Welcome to GERAM CORE OS
              
      
              Everything below runs on this machine. Credentials live in Settings and are never shown here.
              
    
              
    
    
              
      
              What this is
              
      
              A local development environment with two assistants sharing one window.
              
      
              
        
              A.R.E.S.
              Works in your workspace: Monaco editor, file explorer, terminal, source control. It proposes a diff you review and approve — it never edits on its own.
              
        
              I.R.I.S.
              The conversational assistant: voice, reminders, calendar, and the rest of the agents. Optional — the editor works without it.
              
      
              
      
              To get going: 1) add an AI provider key in Settings, 2) open a folder as your workspace, 3) press Ctrl+S to save, like any editor. The full manual is always one click away in the top bar.

        I.R.I.S. · ADAPTED USER MANUAL
        
              Everything you can ask I.R.I.S. to do
              
      
              
      
    
              
    
              
      
      
    
              
    
              
      
              
      
              
        
        
        
        
        
        
        
      
              
      
              
        
              
          01
              Meet I.R.I.S.
              
          
              I.R.I.S. is GERAM's conversational assistant: direct, witty, and built for everyday help. A.R.E.S. is the professional coding role. Both now live inside GERAM CORE OS and can use the AI provider you choose in Settings.
              
          
              Talk naturallyAsk a question or say “system status.” I.R.I.S. answers in the language you use and receives real CPU/RAM data for hardware-status questions.
              
        
              
        
              
          02
              Chat, help, and daily status
              
          
              Chat uses the provider assigned to I.R.I.S. in Settings and replies in the language of your message. Attach a supported image or PDF to ask about it; the file remains local until you explicitly send the message.
              
          
              The microphone sends one bounded recording to a provider that supports audio input. Voice output uses the browser speech engine. Offline mode sends chat directly to the local Ollama service.
              
        
              
        
              
          03
              Voice, camera, and local state
              
          
              VOICE, VISION, and OFFLINE preferences are saved separately for the current operating-system user. VISION requests camera permission, captures one frame, closes the camera, and attaches that image to chat.
              
          
              MIC→transcribe·VISION→attach one frame
              
        
              
        
              
          04
              Files, attachments, and agents
              
          
              Workspace files remain confined to the selected local workspace. PDF text is extracted locally with pdftotext; images are sent only with the chat request that consumes them.
              
          
              Agents shows every bundled module discovered in the installed agent directory plus portable user definitions. “Available” means installed, “loaded” means imported in the running process, and enable/disable state is saved per user.
              
        
              
        
              
          05
              Developer workspace
              
          
              Explorer, Monaco, Source Control, Problems, Preview, and Terminal Watcher form the coding workspace. Python uses Pyright; JavaScript and TypeScript use Monaco language services.
              
          
              <main>becomes<main></main>
              
          
              Tag closing supports HTML, XML, SVG, Vue, Svelte, JSX, and TSX. Emmet expands abbreviations with Tab. A.R.E.S. generates a reviewable diff; approve first, then apply. The secure runner supports Python files, Python unittest, and Node.js .js.
              
        
              
        
              
          06
              Plans, reminders, and connections
              
          
              The Integration Hub has bounded real adapters for Notion, Google Calendar, Telegram, Supabase, Spotify, and an Obsidian vault. Settings reports “connected” only when the required local configuration exists.
              
          
              Integration actions require an agent with the matching permission. Mutating actions also require explicit approval; the presence of a credential never grants an agent access by itself.
              
          
              Settings → AI APIs configures I.R.I.S. and A.R.E.S. OpenAI, Gemini, and Groq are online providers. Ollama is available locally when its service and selected model are installed, and it never requires an API key.
              
        
              
        
              
          07
              Security, privacy, and offline work
              
          
              Workspace paths are validated, sensitive paths are blocked, credentials stay masked, AI edits require review, and code runs through closed sandbox profiles. Preview remains local unless you deliberately start Online sharing.
              
          
              Ollama can provide local chat when its loopback service and selected model are installed. Cloud-backed integrations still require their corresponding connection.
              
          
              Open this manual laterChoose Help → I.R.I.S. Manual at any time. Once dismissed, it will not open automatically again.
              
        
              
      
              
      
              
    
              
    
              
      
              
        
              
          
          
          
          
          
          
          
        
              
        
              
          
              
            01
              Meet A.R.E.S.
              
            
              A.R.E.S. is GERAM's professional development role. It inspects only the files you explicitly place in context, proposes bounded changes, and explains the result in the language of your instruction.
              
            
              Core ruleA.R.E.S. may propose a change, but it cannot approve its own proposal or silently write it into your project.
              
          
              
          
              
            02
              Workspace and editor
              
            
              Use Explorer to open files in the isolated workspace. Monaco provides tabs, autocomplete, diagnostics, Problems, search, Source Control, Preview, and safe file operations. Unsaved edits remain in memory when switching files.
              
            
              Python uses Pyright. JavaScript and TypeScript use local Monaco services. HTML-style tags close automatically in HTML, XML, SVG, Vue, Svelte, JSX, and TSX.
              
          
              
          
              
            03
              Request a code change
              
            
              Open the relevant files, add up to the allowed context limit, and describe one concrete outcome in the inline A.R.E.S. bar. Existing file content is treated as untrusted project data, never as authority to bypass GERAM's rules.
              
            
              Select files→Describe change→Generate proposal
              
          
              
          
              
            04
              Review, approve, and apply
              
            
              Read the unified diff before doing anything else. Approval records the exact proposal digest and file manifest. Applying is a separate explicit action. If a file changes after the proposal was created, GERAM reports a conflict instead of overwriting it.
              
            
              Reject proposals you do not want. Applied multi-file changes use rollback protections if a later write fails.
              
          
              
          
              
            05
              Run and test safely
              
            
              Terminal Watcher exposes three closed execution profiles: Python file, Python unittest, and Node.js .js. Output is bounded and sanitized; jobs have timeouts and can be cancelled. Arbitrary shell commands are not accepted through this runner.
              
          
              
          
              
            06
              Source Control, Problems, and Preview
              
            
              Problems collects diagnostics without sending source code away. Source Control shows repository changes for review. Preview runs locally; Online sharing starts only after an explicit user action and can be stopped from the sharing panel.
              
          
              
          
              
            07
              Boundaries and recovery
              
            
              Protected paths and secrets are excluded from the editor and AI context. Absolute paths, traversal, sensitive files, unsafe symlinks, oversized changes, and stale proposals fail closed. Provider credentials remain masked and outside generated prompts.
              
            
              Open this manual laterChoose Help → A.R.E.S. Manual whenever you need the development workflow again.
